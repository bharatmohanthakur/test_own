## CRITICAL FINDING #1: Evaluation Temperature Changed to 1.0

**Source**: JK-Piece discussion (685920), 15 upvotes, 24 comments

The evaluation **temperature was changed from 0.0 to 1.0**. This was confirmed by JK-Piece (4th place, 0.81) and acknowledged by Kaggle staff (ryanholbrook).

**Impact**:
- Everyone's scores dropped by ~0.03-0.04
- JK-Piece: 0.80+ -> 0.77 on resub
- Yizhou XU (3rd): 0.82 -> 0.80 on resub
- Same submission now gives DIFFERENT scores each time (non-deterministic)
- One user submitted same file 4 times, got: 0.66, 0.67, 0.66, 0.62

**Implications for us**:
- Scores are noisy -- submit same good adapter multiple times and take best
- Train for ROBUSTNESS, not just peak accuracy at temp=0
- Higher temperature means the model needs clearer, more confident reasoning
- Consider training with temperature > 0 during eval/selection

**Status**: Kaggle staff said they are "looking into a rescore" but need RTX GPU availability

## CRITICAL FINDING #2: Training Data Has Major Bugs (50% Poison)

**Source**: Getting Started pinned discussion (681745), comment by user at rank 223

**Data quality issues identified in train.csv**:
- **Bit manipulation**: 50.5% of training traces are WRONG (764/1513 mismatches between computed Result and \boxed{} answer)
- **Equation transformation**: 49% have I/O length mismatches making char_map approach fundamentally broken; 44% have unknown '?' characters
- Pattern: "bit_manipulation is poisoning the model with wrong answers in half its examples"

**Implications for us**:
- NEVER train on raw train.csv data without validation
- Must verify each training example's answer computationally before including
- Data cleaning could be a massive lever -- if top teams cleaned their data, that alone could explain the 0.82 scores
- Consider regenerating CoT traces only for verified-correct examples
- This may explain why 600 curated samples outperform 9500 raw samples

## CRITICAL FINDING #3: Less Data Is Better (600 > 9500)

**Source**: "Train locally and get lower scores" discussion (686419)

Multiple users confirmed:
- 600 samples -> 0.67 LB
- 1200 samples -> 0.66 LB
- 9500 samples -> 0.62 LB (WORSE)

**Why this matters**:
- The train.csv has ~50% bad data in some categories (see Finding #2)
- More bad data = worse model
- Curating a SMALL, VERIFIED dataset is better than using everything
- The 600 samples in the public notebook may accidentally have a distribution closer to test data
- Random seed matters for sample selection

**Action**: Clean the data first, then train on verified subset only

## CRITICAL FINDING #4: Per-Category Error Analysis

**Source**: "Per-Category Error Analysis After SFT" discussion (686069), 8 upvotes

User EnDream (0.63 LB) ran analysis on 300 stratified samples after SFT:
- All 176 errors had properly formatted output (model learned format perfectly)
- Problem is purely **reasoning quality**, not formatting
- **Cipher**: Model generates plausible-sounding text but doesn't do letter-by-letter substitution
- **Gravity/unit_conversion**: Values off by 10-25%, model "estimates" rather than computes
- **Equation transformation**: Consistent with data hallucination

A higher-scoring user (0.68 LB) reported different patterns:
- More trouble with **bit manipulation** puzzles
- Less trouble with cipher & gravity
- Suspects bit manipulation failures stem from **distilled models that yap too much** (Qwen3.5 27B thinking traces)
- Accuracy drops from 49% to 10% when going from 16K to 8K tokens for bit manipulation
- Likely has "poisonous examples" in training data from distillation

**Key insight**: Results vary based on whether you use thinking mode or direct output, and on which teacher model you distill from.

## FINDING #5: GRPO Is Very Hard on Kaggle Resources

**Source**: "RL/GRPO difficulty" discussion (686794)

- RL requires 2 copies of model in GPU memory -> impossible in bf16 on 48GB
- Unsloth doesn't support fast_inference for Mamba hybrid models (GitHub issue #4073)
- vLLM won't work in Unsloth for this model
- No KV cache implemented under transformers inference engine -> painfully slow inference
- Most competitors stuck on SFT approach

**Implication**: If we can get GRPO working, it's a competitive advantage since most people can't

## FINDING #7: NeMo "Train a Reasoning LLM in One Weekend" Recipe

**Source**: NVIDIA Technical Blog

Key recipe details:
- Uses Llama-Nemotron Post-Training Dataset
- LoRA adapter training on single GPU in 48 hours
- For reasoning: 75% reasoning + 25% non-reasoning data mix
- Curate focused subset emphasizing reasoning
- GRPO recipe: train in stages with increasing context (8K -> 16K -> 24K)
- Shorter max_seq_len first helps control long-tail rollout distributions

## FINDING #8: Nemotron 3 Nano Reasoning Details

- Uses `<think>` (token ID 12) and `</think>` (token ID 13)
- 16-bit LoRA fine-tuning uses ~60GB VRAM
- For maintaining reasoning: use at least 75% reasoning / 25% non-reasoning mix
- Synthetic data quality: automated verification (compilers, numerical checks) is critical

## FINDING #9: AIMO-2 Winning Approach (NemoSkills) -- Transferable Lessons

**1st place on AIMO-2** used:
- Qwen2.5-14B-Base fine-tuned on OpenMathReasoning dataset
- Knowledge distillation from DeepSeek-R1 and QwQ-32B
- Three inference modes: CoT, Tool-Integrated Reasoning (TIR), GenSelect
- TIR: model writes Python code for computation
- GenSelect: parallel reasoning chains, select best
- Calibrated training data to match competition format and difficulty

**Transferable to our competition**:
- Distill from strong teacher models (we could use Qwen3.5 27B or DeepSeek-R1)
- Focus on data quality and format matching
- Tool-Integrated Reasoning could be huge for computation-heavy puzzles (gravity, bit manipulation)

## ACTIONABLE NEXT STEPS (Priority Order)

### 1. DATA CLEANING (Highest Impact, Do First)
- Write a validation script to verify each train.csv example
- For bit_manipulation: compute the operation and check against \boxed{answer}
- For equation_transformation: verify char_map consistency
- Remove ALL examples with wrong answers
- Expected: reduce from 9500 to ~4000-6000 clean examples

### 2. BETTER CoT GENERATION (for clean examples)
- Use a strong teacher model (MinMax 2.7 or Qwen3.5 27B) to generate reasoning traces
- Only for examples where the answer is VERIFIED correct
- Keep traces concise (8K token budget for eval means long traces get truncated)
- For bit_manipulation: include step-by-step operation identification
- For cipher: include letter-by-letter mapping derivation
- For gravity: include explicit g calculation from examples

### 3. SUBMIT MULTIPLE TIMES (temp=1.0 exploit)
- Since eval is now stochastic (temp=1.0), submit the same good adapter 2-3 times
- Take the best score
- Save submissions for this strategy

### 4. EXTERNAL REASONING DATA (Broader Generalization)
- Sample from Nemotron-Cascade-2-SFT-Data (math + science subsets)
- Add ~2000-5000 high-quality reasoning examples from external data
- Mix: 60% competition puzzles (verified) + 40% external reasoning

### 5. TRAINING HYPERPARAMETERS
- Current: SFT with 2000 examples, 3 epochs, rank=32, alpha=32
- Try: Clean 600-1200 examples, 3-5 epochs, rank=32, alpha=64
- Consider longer training on smaller verified dataset
- Higher alpha (64 vs 32) for stronger LoRA signal

### 6. GRPO (If Resources Allow)
- Nemotron-Cascade 2 recipe: LR=3e-6, group_size=16, no KL term
- Need to solve VRAM problem (quantized base + LoRA)
- Most competitors can't do GRPO -- competitive advantage if we crack it
