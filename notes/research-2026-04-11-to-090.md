## TL;DR — Top 3 Actionable Levers

### 1. Implement THK's full bit_manipulation algorithm (354-combo scan + left/right sequence matching). Expected +0.10 to +0.13. Effort: 1-2 days.
- THK just revealed (9h ago, discussion 690307) that his 0.85 algorithm solves **1364/1602 = 85.1%** of bit_manipulation (vs Kh0a's 21.88%, m4nocha's 9.23%).
- Full CoT template saved at `/Users/bharat/Downloads/kaggle/notes/thk-bit-cot-full.txt` (845 lines, one complete worked example).
- Algorithm: 21 unary ops (7 ROT + 7 SHR + 7 SHL) × 6 binary ops (AND/OR/XOR and negated variants) × up to 3-transformation compositions; iterates over **pairs of input bit positions** (not expressions) — 18 unary + 336 binary = **354 combos, ~15 tokens each = 5130 tokens total**. Then matches from left and right, picking longest sequence with +1/+1 progression. Fails only on 3-transformation expressions where x+y<8 (output bits depend on 3+ inputs).
- m4nocha's data confirms the arithmetic: 81.34% local with 9.23% bit → THK's 85% with 85% bit adds exactly the missing delta.
- For us: 0.67 → ~0.78-0.80 just by replacing our 70.6% v38 solver with THK's 85.1% one, **assuming** we can train the model to reliably emit 5130-token scan traces. That's the hard part — see blocker #1 below.

### 2. Replicate m4nocha's SFT setup (not GRPO). Expected 0.67 → 0.75-0.79 baseline before bit fix. Effort: 3-5 days.
- m4nocha posted exact config (discussion 689840, 689792): Unsloth 16-bit, `max_model_len=7000`, `max_num_seqs=128`, `gpu_mem=0.9`, inference 1815 tok/s with LoRA. Local 81.34% (LB 0.72 due to test-set distribution). **Was SFT-only, no RL.**
- Their brute-force counts: gravity 100%, unit 100%, numeral 100%, cipher 95.86%, equation 52.11% (numeric-only, 732 problems). Binary 9.23% because they used a weak algorithmic solver. That baseline alone is higher than our 0.67 by ~0.10 **without** even fixing binary or equation symbolic.
- Combine m4nocha's generation config + THK's bit trace → plausible 0.82-0.85 single-adapter submission.

### 3. Switch to NemoSkills-style "simplified recipe" hyperparameters. Expected +0.02-0.04 on top of data improvements. Effort: hyperparameter tweak.
- Confirmed from `NVIDIA-NeMo/Skills/recipes/openmathreasoning/scripts/simplified_recipe.py`: **LR=1e-5, epochs=2, global_batch=32, max_seq=8192, teacher=QwQ-32B temp=0.6 max_tokens=8192**. (Our current: LR=1e-4 with BOXED_LOSS_WEIGHT=5x.)
- NemoSkills LR is **10x lower** than Kh0a's 0.73 notebook. When the dataset is high-quality distillation (vs raw), you want slower LR to preserve base capability. Our drop from v7 (0.69) to v29 (0.52) on templated data is consistent with LR too high for small curated datasets.
- Full NemoSkills pipeline is: QwQ-32B generates solutions at temp=0.6 → filter with ground truth → SFT 2 epochs at LR=1e-5 → eval. **That's it.** No cascaded RL for the Kaggle-focused 14B model.

## Source-by-source findings

### AIMO-2 winning solution (NVIDIA NemoSkills, arXiv 2504.16891)
- Three pillars: (1) OpenMathReasoning dataset 540K problems × 3.2M long-reasoning solutions, (2) Tool-Integrated Reasoning (1.7M TIR solutions), (3) GenSelect for multi-candidate selection.
- **Our competition: only pillar 1 transfers.** No tools allowed, single LoRA, no GenSelect.
- Team solved 34/50 problems in 5 hrs on 4×L4 GPUs with OpenMath-Nemotron-14B-Kaggle.
- Confirmed NemoSkills recipe code: LR=1e-5, max_seq=8192, global_batch=32, epochs=2, sequence_packing enabled, teacher=Qwen/QwQ-32B via trtllm server.
- Data SDG prompt for solutions: `inference.temperature=0.6 inference.tokens_to_generate=8192`, uses `generic/math` prompt config.
- Sources: https://arxiv.org/abs/2504.16891 | https://github.com/NVIDIA-NeMo/Skills/blob/main/recipes/openmathreasoning/scripts/simplified_recipe.py | https://www.kaggle.com/competitions/ai-mathematical-olympiad-progress-prize-2/writeups/nemoskills-1st-place-solution-nemoskills

### AIMO-1 NuminaMath (HF blog)
- Stage 1 CoT: LR=2e-5, batch=32, seq=2048, 3 epochs, cosine, warmup=0.0, **full fine-tune** (no LoRA, explicitly rejected). Data: ~860K OCR'd competition problems translated to English + realigned CoT.
- Stage 2 TIR: LR=2e-5, batch=32, seq=1024, 4 epochs, warmup=0.1. Data: 60K problems × GPT-4-generated ToRA-style code execution traces, filtered by final-answer match, 3 iterations.
- Single-LoRA-no-tools ceiling: **56.3% MATH bench (NuminaMath-7B-CoT)**. TIR adds 12 percentage points (68.2% MATH) via tool execution which we can't use.
- Source: https://huggingface.co/blog/winning-aimo-progress-prize

### Nemotron-Cascade (nemotron research)
- Order: SFT → RLHF → IF-RL → Math RL → Code RL → SWE RL. Each subsequent stage rarely degrades prior benchmarks.
- Math RL dataset: `nvidia/Nemotron-Cascade-RL-Math` = 14,476 problems from OpenMathReasoning + NuminaMath-CoT + DeepScaleR + AceReason-Math.
- **Not applicable**: multi-stage full RL on 30B+ model, needs far more than single 96GB GPU. Our single-LoRA budget can do at most SFT + one light GRPO pass.
- Source: https://research.nvidia.com/labs/nemotron/nemotron-cascade/

### Komil Parmar's GRPO speed fix (discussion 690161, 17 upvotes)
- Root cause: `prepare_inputs_for_generation` sends cache as `past_key_values`, `forward` expects `cache_params`. Mismatch → cache silently dropped → 2 tok/s vs 38 tok/s.
- Fix: **transformers>=5.3.0** (confirmed 5.5.3 works) + **drop `trust_remote_code=True`** (or old modeling file overrides fix) + **`gradient_checkpointing=False`** (native impl doesn't support it).
- This is 20x GRPO speedup. A 500-step GRPO run goes from days to hours.

### Donald Galliano III playbook (discussion 688461, 110 upvotes)
- Full 3-input gate list confirmed: **MAJ, CHO, PAR3, AO, OA, AX, OX, XA, XO**.
- Full 4-input list: **AOA, OAO, PAR4, XX, AXA**.
- Key insight: bit-serial gate computation (0&1=0 1&1=1 ...) is **non-negotiable** — parallel blob computation craters accuracy to 9.3%. This matches m4nocha exactly.
- VER (target verification on actual test input, not examples) is also non-negotiable — multiple ops can coincidentally match 8 example columns.
- Per-bit GRPO reward with partial credit at each layer turns it from a cliff into a gradient.

### m4nocha (132nd) config (discussions 689792, 689840)
- Unsloth GRPO working; 2 hrs/epoch for 3000-token seq.
- Uses vLLM for post-SFT eval: `max_model_len=7000, max_num_seqs=128, gpu_memory_utilization=0.9, enable_lora=True`.
- **Per-category validation on 950 samples**: gravity/unit/numeral 100%, cipher 95.86%, equation 52.11%, binary 9.23%, symbol 0% → overall 81.34% local, 0.72 LB.
- Confirmed LB ≈ local − 0.09 on their setup (distribution skew).

## Blockers making 0.90 unreachable with current constraints

1. **7680 token generation budget**: THK spends 5130 tokens just on the binary scan. A second puzzle type in the same prompt + thinking overhead doesn't fit. 0.90 would require solving BOTH symbol_transform cipher-digit (5000+ tokens) AND 3+ input gate binary (3000+ tokens) — but you only get one completion per problem. Hard cap.
2. **Hidden test set distribution**: If hidden test has more symbol_transform (m4nocha/Kh0a 0%) or broader math (we have zero broad math data), ceiling is <0.85. James Day speculated 0.90 is "the ceiling" and he has an ensemble (can't submit). THK confirmed 250-puzzle LB, so each puzzle = 0.004 — 0.85 vs 0.90 is only 13 puzzles.
3. **Symbol_transform genuinely unsolved publicly**: Nobody in the top 50 has posted a symbolic equation solver. THK likely has it but won't share until Sunday. Without it, hard ceiling at ~0.85 (100% × 5 categories + 85% binary + 0% symbol ≈ 0.83-0.85 depending on symbol proportion).
4. **Model can't do algorithms with low data**: We'd need 10K+ high-quality bit traces to reliably train THK's 354-combo scan. Our 9500 is borderline, and the competition's 5-submission/day rate-limits the feedback loop.
5. **Midpoint prize already locked (April 9)**: THK's Sunday publication will reveal everything. We should **wait for Sunday release (12 hours)** before committing v39 compute — their code will be far better than my reimplementation.
