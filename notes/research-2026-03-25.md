## 1. JK-PIECE'S APPROACH (CONFIRMED DETAILS)

### Published Dataset: "AIMO3 Tool-Integrated Reasoning"
- **URL**: https://www.kaggle.com/datasets/jeannkouagou/aimo3-tool-integrated-reasoning
- **Size**: 141,277 training examples (deduplicated)
- **Generator**: GPT-OSS-120b using "Harmony protocol"
- **Format**: CSV with columns: prompt, completion, problem_id, source, license
- **Key Innovation**: Real Python code execution traces from a stateful Jupyter kernel (NOT synthetic traces)
- **Solution Hint Methodology**: Complete solutions provided as subtle guides during generation to ensure correctness, then hints removed from final training data
- **All traces contain**: step-by-step reasoning + Python tool calls + execution results + verified answers in \boxed{} format
- **272 downloads, 1147 views, 24 upvotes** — widely used by other competitors

### What This Tells Us About JK-Piece's Strategy
1. **Tool-Integrated Reasoning (TIR)** is the key differentiator — model learns to write AND execute Python code as part of reasoning
2. **Scale**: 141K examples is 15x our 9,500 training set
3. **Quality guarantee**: Solution hint methodology ensures correctness (reject bad traces)
4. **Olympiad-level difficulty**: All problems from math olympiad competitions
5. **Format conversion utilities**: Includes convert_harmony_format.py for ChatML/Llama/Anthropic formats

### Likely Full Approach (Inferred)
- SFT on 141K TIR traces to teach the model to use Python as a tool
- Possibly additional puzzle-specific data for the 6 competition puzzle types
- Possibly GRPO on top of SFT for efficiency and accuracy refinement
- The Harmony protocol format suggests training the model to produce structured tool calls

## 2. KEY TECHNIQUE: TOOL-INTEGRATED REASONING (TIR)

### Why TIR Creates a Massive Accuracy Gap
- **TIR models verify reasoning through computation** — instead of just thinking, they compute and check
- **TORL-7B achieved 43.3% on AIME24** — surpassing RL without tool integration by 14% and best existing TIR model by 17%
- **NuminaMath-7B-TIR** won AIMO-1 by decomposing each problem into rationale + Python code + execution output
- **The format**: Each problem solved via interleaved natural language reasoning and Python REPL calls

### How This Applies to Our Competition
The 6 puzzle types (cipher, bit manipulation, equations, gravity, unit conversion, numeral systems) are ALL programmatically solvable. A TIR-trained model would:
1. Read the puzzle description
2. Write Python code to parse and solve it
3. Execute the code
4. Verify the answer
5. Output the final answer

**This is likely the single biggest lever for closing the 0.69 -> 0.81 gap.**

### Implementation Plan for TIR
1. **Generate TIR training data**: For each of the 9,500 puzzles, create traces showing:
   - Natural language analysis of the puzzle
   - Python code that solves it step by step
   - Execution output
   - Final verified answer
2. **Use JK-Piece's dataset format** as a template (Harmony protocol -> ChatML conversion)
3. **Add puzzle-specific Python solvers** for each type as code execution demonstrations

## 3. TWO-STAGE RECIPE: SFT THEN GRPO (THE PROVEN WINNING APPROACH)

### Research Paper: "A Practical Two-Stage Recipe for Mathematical LLMs" (arxiv 2507.08267)
**Key finding**: SFT and GRPO play complementary roles:
- **Stage 1 (SFT)**: Push accuracy to its maximum — extend SFT for as many as 10 epochs
- **Stage 2 (GRPO)**: Optimize token efficiency while preserving peak accuracy
- GRPO's primary role is to make the model produce shorter, more efficient solutions

### "Improving DeepSeek R1 in Math" Blog (hav4ik.github.io)
**Concrete results**:
- 14B model: 75.8% Maj@32 on AIME'25 (+8.7% improvement from SFT+GRPO)
- 7B model: 65.8% Maj@32 (+7.5%)
- **SFT dataset**: 10K samples — 8K from NuminaMath-1.5 + 2K from Light-R1
- **GRPO dataset**: Harder half of the SFT dataset
- **Training**: 8xH100, 6 epochs, 16K context length
- **Framework**: Open-R1 with faster GRPOTrainer, veRL fork with DAPO + FSDP-LoRA + Dr. GRPO

### GRPO Improvements Over SFT Baselines (DeepSeekMath)
- GSM8K: 82.9% -> 88.2% (+5.3%)
- MATH: 46.8% -> 51.7% (+4.9%)
- **This 5-10% gain from GRPO on top of SFT could explain part of the 0.69->0.81 gap**

## 4. LORA + GRPO ENGINEERING DETAILS

### Tina: Tiny Reasoning Models via LoRA (ICLR 2026)
- **Base**: DeepSeek-R1-Distill-Qwen-1.5B
- **Method**: LoRA during RL with GRPO-style algorithm
- **Result**: >20% reasoning performance increase, 43.33% on AIME24
- **Cost**: Only $9 USD
- **Key insight**: LoRA rapidly adapts the model to the structural format of reasoning rewarded by RL, while largely preserving base model knowledge

### GRPO + LoRA with Verl (HuggingFace Engineering Handbook)
- **URL**: https://huggingface.co/blog/Weyaxi/engineering-handbook-grpo-lora-with-verl
- **Setup**: Qwen2.5-3B-Instruct, TP=4 across GPUs
- **Results**: Validation accuracy 59% -> 85% in 100 steps
- **Key config**: trainer.total_epochs can be cut from 15 to 5 for 3B models
- **Convergence**: LoRA converges faster than full fine-tuning while being more VRAM-efficient

### RL vs SFT Efficiency Gap (Critical Insight)
- RL is fundamentally more efficient than SFT at low parameter counts
- SFT requires 100-1,000x more parameter updates to reach same performance as RL
- SFT forces model to absorb stylistic noise; GRPO provides sparser but cleaner signal
- **Implication**: For our competition with rank-32 LoRA cap, GRPO may be MORE effective per-parameter than SFT alone

## 5. NVIDIA'S OWN RECIPE (Nemotron 3 Nano Training)

### From the Technical Report + Blog
1. **SFT Phase**: Fine-tune on mix of reasoning (75%) and non-reasoning (25%) data across math, code, science, tool calling, instruction following
2. **RL Phase**: Synchronous GRPO across multiple environments (math, code, science, instruction following, multi-step tool use)
3. **Key hyperparameters**:
   - LR: 1e-4 with cosine decay, 5% warmup
   - LoRA rank: 64+ recommended (we're capped at 32)
   - Alpha: 128 for rank 64 (ratio 2:1)
   - Batch size: 256 effective
   - Steps: 2,000+ minimum
   - Context: Match eval context length in training
4. **Do NOT fine-tune router layers** in MoE — disabled by default in Unsloth
5. **Target modules**: q_proj, k_proj, v_proj, o_proj (Unsloth recommendation for Nemotron)

### From NVIDIA "Train in One Weekend" Blog
- LoRA rank of at least 64 with alpha = 128 recommended
- 48 hours on single H100 for 2000 steps
- Consistent reasoning emerged after ~13 hours
- Data: ~500,000 balanced samples recommended
- Curated dataset from Llama Nemotron Post-Training Dataset

### Breakdown of Expected Gains

| Technique | Expected Gain | Cumulative |
|-----------|--------------|------------|
| Current SFT baseline | 0.69 | 0.69 |
| + Tool-Integrated Reasoning data | +0.04-0.06 | 0.73-0.75 |

### The Three Pillars of the Gap
1. **Data Quality & Format (60% of gap)**: TIR format with code execution traces vs plain CoT. JK-Piece's 141K high-quality TIR examples vs our 9,500 basic SFT examples.
2. **Training Method (25% of gap)**: SFT alone plateaus. GRPO on top provides 5-10% additional gains by optimizing the reasoning strategy.
3. **Configuration (15% of gap)**: Better LoRA targets, alpha ratio, context length matching, curriculum learning.

WINNING = TIR Data (code execution traces)
        + Scale (50K-141K examples)

        + Proper LoRA Config (all modules, alpha=2*rank)

The #1 competitor (JK-Piece at 0.81) almost certainly uses Tool-Integrated Reasoning as the core technique, with 141K+ training examples, and likely applies GRPO or similar RL on top of SFT.
