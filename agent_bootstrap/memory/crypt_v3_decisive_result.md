**2026-06-07.** crypt_v3 trained on 185 short, DECISIVE, agent-reasoned verification traces
(generated by Claude subagents on the Claude Code backend, median ~163 tok, 100% boxed==truth,
verify-once-then-commit style). Fresh-from-base r=32, batch 4 x accum 8, 8 epochs, max_seq 7680.

**Bench (vLLM, temp=0, 7680 tok, n=30): 3/30 = 10.0%.**

Three crypt SFT attempts now, all clustered low:
| version | data | acc | termination |
|---|---|---|---|
| v1 | 727 terse solver traces | 16.7% | finishes fast, WRONG answers (hasty) |
| v2 | 309 long Sonnet verification | 3.3% | 27/30 NEVER terminate (loops to 7680) |
| v3 | 185 short decisive agent traces | 10.0% | 19/30 terminate; of those only 3 correct |

**Key positive finding:** the decisive/short style FIXED v2's non-termination — 19/30 finish
with a boxed answer (vs v2's 3/30), avg gen tokens 7193 -> 3046, ceiling-hits 27 -> 8. So
trace LENGTH/decisiveness controls termination. But terminating more just exposed v1's other
failure (finishes but gets the wrong rule). v3 ~= v1 statistically (3 vs 5 of 30 = noise).

**Conclusion: SFT on worked-example traces — terse, verbose, or decisive — does NOT crack crypt.
Ceiling is ~10-17%.** Crypt is a constraint-SEARCH task the 30B model cannot learn to EXECUTE
greedily from examples. Consistent with [[crypt_dpo_v2_solver]] (DPO 2/10), [[v40_pathA_result]]
(embed_tokens), GRPO. The lever is NOT more/better SFT traces.

**Realistic options going forward:**
1. Accept crypt ~= v26's level (~2/10); do NOT regress the other 5 types. Ship v26-class.
2. Tool-use at INFERENCE: expose the deterministic CSP solver (scripts/gen_crypt_decisive.py
   logic / crypt_solver.py) as a tool the model calls — bypasses the un-learnable search.
3. Stop iterating crypt SFT — diminishing returns proven 4x.
